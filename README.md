# Station Throughput Lab

Predicting next-day bike-station throughput with geospatial features, cold-start imputation, and calibration-based bias correction.

A project exploring how to forecast demand at individual service nodes in an urban network — including early-life stations with sparse history and forecast periods with different demand characteristics than training.

## The Problem

Bike-share stations have very different usage patterns depending on location, day of week, weather, and neighborhood context. Predicting next-day departures per station is useful for rebalancing operations. Two hard parts:

1. **Cold start** — new stations can have sparse or unavailable station-specific history
2. **Distribution shift** — demand patterns change seasonally, and the model must generalize from training months to a different test period

## Approach

### Feature Engineering
- **Temporal:** day-of-week (cyclical), month, holiday/weekend, lag features (1d, 2d, 7d, 14d, 28d), rolling mean/median/std (7/14/28d), DOW-specific rolling mean and median, trend ratios, coefficient of variation, zero-day rate
- **Geospatial:** DBSCAN spatial clusters, distance to city center, station density within 1km, k-nearest-neighbor average throughput, borough proxy
- **Weather:** NOAA Central Park daily observations when available (temperature, precipitation, snow), weather × temporal interactions (cold_weekend, rain_weekend, temp_deviation)
- **Station maturity:** age in days, new-station indicators, historical average throughput (capacity proxy), weekend-vs-weekday ratio
- **Network-level:** city-wide and cluster-level demand signals (yesterday's total, rolling averages)

### Cold-Start Imputation
Hierarchical fallback for stations with insufficient history:
1. **Spatial cluster average** — throughput of mature stations in the same cluster, scaled by local density ratio
2. **Borough average** — fallback when cluster data is sparse
3. **City-wide average** — final fallback

### Modeling
AutoGluon TabularPredictor with `best_quality` presets and dynamic stacking disabled for local reproducibility. The framework handles model selection and ensembling. The work is in the features.

The evaluation is a rolling one-day-ahead setup: lagged station, neighbor, and network features use demand observed through the previous day.

### Calibration & Bias Correction
Post-model hierarchical multiplicative correction estimated from the calibration set:
1. **Station-level** factors (most specific)
2. **Cluster × DOW** factors
3. **Borough × DOW** factors
4. **Global** factor (fallback)

This separates signal learning (model) from level correction (calibration), a common pattern in operational demand planning.

### Distribution Shift Analysis
JS divergence between train and test feature distributions identifies which features drifted and explains the raw model's bias. December temperature shifts severely and precipitation shifts moderately; lag features and rolling demand patterns remain stable.

## Data

NYC Citi Bike public trip data:
- https://citibikenyc.com/system-data
- ~40M+ trips across 9 months, 2,200+ stations after filtering
- Period: April 2024 – December 2024
- Weather: NOAA daily summaries for Central Park, with deterministic fallback if unavailable

## Key Results

| Metric | Raw Model | After Calibration |
|--------|-----------|-------------------|
| Test MAE | 11.70 | **10.79** |
| Test WAPE | 0.335 | **0.309** |
| Test MAPE (nonzero actuals) | 0.582 | **0.530** |
| Test Bias | +0.149 | **+0.038** |

- MAPE is computed only on rows with nonzero actual departures; WAPE remains the primary percentage-error metric
- Early-life stations: post-calibration MAE gap ~28.0% higher than mature, with more uncertainty than established stations
- Top features: dep_lag_1, rolling_mean_7, DOW-specific rolling averages
- 1 of 15 top features shows severe drift (temperature); lag and rolling demand features remain stable by JS divergence

See `reports/throughput_lab_report.md` for the full analysis.

## Quick Start

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt

# Full run (~5 min download + 20 min training)
PYTHONPATH=src python -m station_throughput_lab.run_project

# Shorter model-fit pass after data/features are cached
PYTHONPATH=src python -m station_throughput_lab.run_project --time-limit 120
```

Large generated artifacts are intentionally excluded from version control:
raw trip archives, processed parquet caches, trained model files, and the
row-level scored panel can all be regenerated by the pipeline.
